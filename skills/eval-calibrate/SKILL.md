# eval-calibrate — 評価するAIを較正する

評価スキルは最初は間違える（AI評価の精度はガードレール水準から始まる）。それでも成立するのは、間違いが較正データとして回収され、次の評価を1つ賢くする構造があるから。このスキルがその回収装置。

## 手順

### 1. 突き合わせ
AI評価JSON（calibration/の該当ファイル）と人間FBを観点単位で照合し、3分類する。**一致判定は帯（1–3/4–7/8–10）で見る** — ±1点のズレを追いかけると較正が収束しない:
- **一致**: AIと人間が同じ帯 → 精度カウンタ加点
- **見落とし**: 人間は問題視したがAIはpass帯にした/観点自体が無かった
- **誤指摘**: AIは低い帯にしたが人間は問題なしとした
class Cの**影スコア**も同様に帯一致を記録する（集計外だが較正対象としては同格）。影スコアの帯一致率が安定して高くなった観点は、B昇格（配点対象化）の提案を出す。
未確認テーブルに載っていた観点への人間指摘は「見落とし」ではなく「未確認由来」として別カウント。理由: 見なかったのと見て外したのでは直す場所が違う（前者は入力・手順の問題、後者は基準の問題）。

### 2. カウンタ更新とルール発動
観点ごとの精度カウンタ（一致/見落とし/誤指摘の累計）を更新し、しきい値で提案を出す:
- **同一観点で見落とし2回** → 基準文の追加、または新観点の起票を提案（検証が1つ増える）
- **同一観点で誤指摘2回** → 基準文を絞る提案（ルールが正確になる）
- **どの観点にも該当しない人間指摘** → 新観点の起票を提案
- **B観点でAI判定が人間と安定一致＋機械照合化の目処** → Aへの昇格を提案
- **B観点で誤指摘が減らない・確信度が上がらない** → Cへの降格を提案（AIは観察列挙まで）

### 3. 承認ゲート
**提案止まり。基準の書き換え・昇格降格の実行は人間の承認後。** 理由: AIが自分の採点基準を自分で書き換えられると、較正が「人間に合わせる」ではなく「自分の過去の判定に合わせる」方向に閉じる。ルールを増やしすぎるノイズも人間が止める。

### 4. 記録
突き合わせ結果を `calibration/YYYY-MM-DD_<案件ID>_calibration.json` に保存: 3分類の明細・カウンタ現在値・発動した提案・人間の裁決。
配点と人間の総合評価がズレた場合（点数は高いのに人間の評価が低い等）は、観点の判定ではなく**配点プロファイルの修正候補**として別記する。

## Gotchas
- 人間FBの「なんか違う」をそのまま観点化しない。どの観点の話かを人間に1問だけ確認してから分類する（推測で振り分けると較正が濁る）。
- 討論モデルの乖離点は優先的に突き合わせる。人間がどちらに軍配を上げたかは専門家スキルの基準文を直す最良のデータ。
- 誤指摘の較正で基準を絞りすぎない。絞った直後の2〜3件は見落とし側の増加を監視する（振り子の打ちすぎ防止）。
- カウンタはリセットしない。基準を直したら「直した日付」を記録してそこから再カウント — 直す前の成績と混ぜると改善効果が測れない。

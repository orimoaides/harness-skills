# banner-eval — バナー評価の実行体

評価は3段階で行う。順番を守る理由: 後段ほどコストが高く主観が混ざるので、機械で白黒つくものを先に落とせば、定性判定を「本当に判断が必要なもの」だけに絞れる。

## 手順

### 0. 入力の確認
必要なもの: ①評価対象の画像 ②ブリーフ（媒体・サイズ・必須要素・NG事項・ターゲット・訴求優先順位）③ブランドプロファイル（`references/brand-profile.md`）④評価セット（eval-orient の出力。無ければ全観点）。
ブリーフが無い場合、ブリーフ適合系の観点はすべて「未確認」として扱う。**推測でブリーフを補完しない** — 補完した瞬間、評価が依頼ではなく想像に対するものになり、較正データとして無価値になるため。

### 1. class A — 機械照合（ブロックゲート）
`references/class-definitions.md` のA観点を照合する。判定は true/false のみ。
- 1つでも false → **点数をつけずに差し戻し**。理由: A違反（必須要素の欠落・誤字・サイズ違反・ロゴ規定違反）は程度問題ではなく、直すまで他の議論に意味がないため。
- Aは意図宣言（「あえてこうした」）でも覆らない。変更したければブリーフ側を更新するのが筋。
- 照合できないA観点（規定文書が未提供など）は false ではなく **unverified** として報告する。falseにすると「規定が無い」と「規定に違反した」の区別が消え、較正できなくなる。

### 2. class B — 定性判定（3段階）
評価セットに含まれるB観点を1つずつ判定する。出力は観点ごとに:
`{ 観点ID, 判定: pass|minor|fail, 根拠(1-2文), 測定手段: 実測|目視|読解, 確信度: high|mid|low }`
- 測定手段は必ず書く。理由: 「目視」の判定は較正時に人間が優先的に疑うべき箇所であり、どれが目視かを隠すと較正が効かない。
- 迷ったら fail ではなく **minor＋確信度low**。理由: 評価専任者が過剰にfailを出すと誤指摘カウントが濁り、基準を絞る較正が誤作動する。
- class C観点（人間領域）は判定しない。「観察事実の列挙」のみ（例: 「見出しの字間が広く見える〔目視・low〕」）。

### 3. 採点 — 集計は計算、判断はしない
`references/scoring.md` の配点表に従い、pass=満点 / minor=半分 / fail=0 で機械集計する。
- **合計点をAI自身の感覚で丸めない。** 理由: 点数の価値は人間FBとの差分を測れることにあり、感覚で調整した点は較正に使えない。
- craft（出来栄え点）と compliance（A照合の結果）は別欄。高得点でもcomplianceがNGならNG。

### 4. 出力
1. **JSON**（正本）: `references/output-format.md` のスキーマで `calibration/` に保存。**未確認テーブル**（確認できなかった観点＋理由）を必ず含める。理由: 「見てない」を「問題なし」と区別しないと、見落としと未確認の較正が混ざる。
2. **HTML**（人間向け）: JSONからペラいち自己完結HTML（画像埋め込み・観点と判定を同じ行に・専門家間乖離があれば明示）を生成。

## Gotchas
- 画像は必ず実寸で確認する。縮小プレビューだけで判定すると、にじみ・エッジ・小さい文字の破綻を見逃す（実績あり）。
- 「文字入りの完成物」は文字組みに重み4割の変則配点に切り替える（scoring.md参照）。切り替え忘れると文字問題が軽く出る。
- 評価対象が複数案あるときは、案ごとに独立して評価してから比較する。最初から比較で始めると相対評価に引っ張られ、単体の絶対判定が歪む。
- このスキルは**直さない**。改善指示は書くが、修正版の生成はしない。検品員が直し始めると、自分の修正に甘い判定を出すようになるため。

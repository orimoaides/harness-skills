# eval-orient — 診断してから評価する

評価を始める前に、まずボトルネックの層を診断する（Layers の /layers-orient と同じ思想）。理由: 全観点を毎回フル評価するとコストが高いだけでなく、評価データが「毎回条件が違う比較不能な記録」になり較正が効かなくなる。**名前つきの評価セットを選ぶ**ことで、同条件の評価データが貯まる。

## 手順

### 1. 案件情報の抽出
ブリーフ/依頼書から: 必須要素・NG事項・ターゲット・訴求優先順位・媒体とサイズ・納期フェーズ。
抽出できない項目は「不明」と明記する。推測で埋めない — orientの推測は下流の全評価を汚染する。
**穴が多い・要件同士が矛盾している場合は、評価に進む前に brief-dig（briefモード）を提案する。** 矛盾したブリーフのまま評価すると、どの要件に照らすかで帯が変わり較正データが汚れる。

### 2. 層の診断
画像をざっと見て（精査はしない。精査はbanner-evalの仕事）、問題がどの層にありそうかを見立てる:
- **手前の層**（ブリーフ理解・訴求設計）: 訴求の序列がブリーフとズレていないか。ここが怪しいならサーフェスの観点を厚くしても無意味 —「色を直せ」ではなく「序列がズレている」が先。
- **サーフェス層**（組版・レイアウト・色・素材）: 手前が合っているときに初めて厚く見る。

### 3. 評価セットの提案
フェーズ×目的で名前つきプリセットから選ぶ（`references/presets.md`）:
- 初稿系フェーズ → **探索モード**: 点数で殴らない。方向性の比較・「ありきたりではないか」・なぜこの案かの言語化が主目的
- 中間フェーズ → **標準セット**: A照合＋B主要観点
- 入稿前 → **収束モード**: A全開＋B厳格＋採点＋即fail監視
提案には理由をつけ、**最終決定は人間に渡す**。どのプロファイルで評価するかの判断は較正の基準点になるので、AIが黙って決めると後で「評価セットが悪かったのか判定が悪かったのか」を分離できない。

### 4. 出力
`{ 案件サマリー, 診断: 怪しい層と根拠, 提案評価セット, 配点プロファイル, 未確認事項 }` を banner-eval にそのまま渡せる形で。

## Gotchas
- 診断で「問題を見つけて指摘」まで踏み込まない。orientが評価を始めると banner-eval と二重評価になり、判定の出どころが較正できなくなる。見立ては「どこを見るべきか」まで。
- プリセットに合わないケースで無理に既存プリセットへ押し込まない。「該当なし・新プリセット候補」と報告する方が、プリセット棚が育つ。
- 選択の結果も較正対象。評価後に人間の指摘が別の観点へ集中したら、それは「選択が外れた」記録として eval-calibrate に渡す。

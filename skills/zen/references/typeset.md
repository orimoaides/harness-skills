# typeset — 日本語タイポグラフィ評価モード

`/zen` のタイポグラフィモード。日本語Webサイトのタイポグラフィを評価し、具体的な改善を行う。

英語タイポグラフィのルール（`65-75ch`、`line-height: 1.5` 等）をそのまま日本語に適用すると破綻する。このモードは日本語固有の組版知識に基づいて判断する。

## ワークフロー

### Step 1: 現状の読み取り

まず対象サイトの用途を判定し、プロファイル（media / saas / docs / dashboard）を決める。プロファイルによって推奨値が変わる（リファレンスのセクション11を参照）。

対象が指定されている場合はそのファイルを読む。指定がなければ以下を探す:

1. グローバルCSS（`globals.css`, `global.css`, `app.css` 等）
2. Tailwind設定（`tailwind.config.*`）
3. フォント設定（`next/font` のインポート、`@font-face` 宣言）
4. レイアウトファイル（`layout.tsx`, `_app.tsx` 等）
5. 主要なテキストコンポーネント

読み取る観点:
- 使用フォント（日本語・英語）
- フォントサイズ体系（モジュラースケールに乗っているか？恣意的な値がないか？）
- 行間（line-height）
- 行長（max-width）
- 余白の体系（Vertical Rhythm / 8の倍数 / Modular Scale / なんらかの規則があるか？）
- 字間（letter-spacing）
- font-feature-settings
- ウェイト構成
- 禁則処理の設定

### Step 2: 評価

[日本語タイポグラフィ リファレンス](japanese-typography.md) に基づいて評価する。

以下の各項目を ✅ / ⚠️ / ❌ で判定:

**体系性（ハーモニーとリズム）** ← 最も重要な観点
- 文字サイズがモジュラースケールに乗っているか（一つの比率から導出されているか）
- スケール外の恣意的な値（17px, 22px 等）が混在していないか
- 余白に規則性があるか（Vertical Rhythm / 8の倍数 / Modular Scale / フォントサイズベース）
- 行送り（line-height × font-size）が余白体系の基準単位になっているか

**行長**
- **長文コンテンツ**（記事、ブログ、ドキュメント）のコンテナが `35-45em` の範囲内か
- `ch` 単位を使っていないか（日本語では不適切）
- LP・カード・UI要素に行長制限を誤って適用していないか（レイアウトが壊れる原因になる）

**行間**
- 本文の line-height が 1.8-2.0 か
- 見出しの line-height が適切に狭められているか
- ダークモードで line-height の補正があるか（任意）

**フォント選定**
- 日本語フォントが明示的に指定されているか
- Windows 游ゴシック問題に対処しているか（Medium 指定 or フォールバック）
- フォントペアリングに意図があるか
- Webフォントの読み込み最適化（`font-display: swap`、ウェイト数）

**和欧混植**
- 日英フォントの指定順が適切か（英語→日本語）
- `chws` / `vchw` の OpenType feature が有効か（推奨）
- サイズ感のバランスが取れているか

**約物**
- `palt` / `halt` の適用が適切か
- 見出しと本文で使い分けているか

**文字サイズ**
- 本文が 16px (1rem) 以上か
- `rem` 単位を使っているか
- タイプスケールに一貫性があるか

**ウェイト**
- 使用ウェイト数が3-4以内か
- 日本語での太さ感を考慮した使い方か
- Bold の多用がないか

**禁則処理**
- `overflow-wrap: break-word` が設定されているか
- `line-break: strict` が設定されているか（推奨）

**Webフォント読み込み**
- `font-display: swap` が設定されているか
- 読み込むウェイト数が適切か（3以内推奨）
- Next.js なら `next/font` を使っているか

**Reject / Warn チェック**

判定条件の一次定義は [日本語タイポグラフィ リファレンス](japanese-typography.md) セクション12（R1–R6 / W1–W6）。critique を含めた他モードからも同じ条件を参照するため、ここでは再列挙しない。Reject に該当する問題は最優先で報告する。

### Step 3: 報告

スコア算出: ベース 10 点から Reject 1 件ごとに -3、Warn 1 件ごとに -1 を減算。下限 1 / 上限 10。評価対象が存在しない項目（例: Webフォント未使用のサイトで `font-display` 指摘不可）は減点対象外。

評価結果を以下の形式で報告する:

```
## 日本語タイポグラフィ評価

### スコア: X / 10

✅ 行長: 45em で適切
⚠️ 行間: 本文 1.5 → 1.8 に広げるべき
❌ フォント: 游ゴシック問題が未対処
...

### 改善提案（優先度順）

1. **[高]** ...具体的なコード変更...
2. **[中]** ...
3. **[低]** ...
```

### Step 4: 修正

ユーザーが修正を求めた場合、具体的なコード変更を提案・実行する。

変更時の原則:
- 既存のデザイントークン / CSS変数がある場合はそれに従う
- Tailwind を使っているプロジェクトでは Tailwind のユーティリティクラスで表現
- 変更は最小限に。タイポグラフィ以外のレイアウトを壊さない
- 変更前後でどう見え方が変わるかを説明する

## やらないこと

- 配色の評価（このモードの範囲外）
- レイアウト・余白の全般的な評価（行長に関連する場合のみ言及）
- コンテンツの推敲（テキストの中身には触れない）
- フォントの購入・ライセンス判断（情報提供のみ）

# 日本語UXコピー リファレンス

UIに現れる短いテキスト — ボタン、エラーメッセージ、フォームラベル、通知、ツールチップ — を対象とする。ブログ記事やマーケティングコピーは対象外。

## 0. 設計思想 — 明確・一貫・温かい

UIコピーの仕事は、ユーザーが迷わず、不安にならず、次の行動に移れるようにすること。

### 明確

ユーザーが読んで1秒で理解できるか。曖昧な表現、省略しすぎた主語、二重否定は排除する。

- ❌「処理に失敗しました」→ 何が失敗したのか不明
- ✅「ファイルを保存できませんでした。空き容量を確認してください」

### 一貫

同じ操作には同じ言葉を使う。一画面内で敬語レベル・動詞活用・表記が揺れると、ユーザーは無意識に違和感を感じる。

- ❌ 「送信」「キャンセルする」「戻る」が同じ画面に混在
- ✅ 全ボタンを「送信する」「キャンセルする」「戻る」（辞書形で統一）or「送信」「キャンセル」「戻る」（体言止めで統一）

### 温かい

エラーや制約を伝えるとき、ユーザーを責めない。システムの都合を押し付けない。

- ❌「不正な入力です」→ ユーザーを「不正」と呼んでいる
- ✅「入力内容を確認してください」

ただし過度に丁寧にしない。「大変申し訳ございませんが、ご入力いただいた情報に誤りがある可能性がございます」は冗長。

## 1. 敬語レベル体系

UIコピーの敬語は4段階ある。プロダクト全体で一つのレベルに統一する。

### 4段階

| レベル | 文末 | ボタン例 | 適するプロダクト |
|---|---|---|---|
| カジュアル | 〜する / 〜だ | 送信する、削除する | 個人向けアプリ、ゲーム、SNS |
| です・ます | 〜します / 〜です | 送信します、削除します | SaaS、一般的なWebサービス |
| 丁寧 | 〜してください / 〜ください | 送信してください、ご確認ください | BtoB、金融、医療 |
| 最丁寧 | 〜いただけますか / 〜くださいませ | ご送信いただけますか | 高級ブランド、公官庁（UIにはほぼ不適） |

### 選び方

- **ボタンラベル**にはレベルを反映しない（体言止めか辞書形が多い）
- **説明文・ヘルプテキスト**でレベルが決まる
- **エラーメッセージ**は説明文より半段階丁寧にするのが安全（失敗時は寄り添う）
- 一つの画面でレベルを混在させない

### よくある混在パターン

```
⚠️ 混在例:
ボタン:「送信」（体言止め）
フォーム説明:「名前を入力する」（辞書形 — カジュアル）
エラー:「正しい値をご入力ください」（丁寧）
→ 3レベルが1画面に混在
```

```
✅ 統一例（です・ます調）:
ボタン:「送信する」
フォーム説明:「名前を入力してください」
エラー:「メールアドレスの形式を確認してください」
```

## 2. ボタンラベル

### 活用形式の4パターン

| 形式 | 例 | 印象 | 使いどころ |
|---|---|---|---|
| 体言止め | 送信、削除、保存 | 簡潔、テック的 | ダッシュボード、管理画面、アイコン付きボタン |
| 辞書形 | 送信する、削除する | 自然、動作感 | 一般的なWebアプリ、SaaS |
| ます形 | 送信します、削除します | 丁寧、フォーマル | BtoB、金融 |
| 文脈付き | この内容で送信する、アカウントを削除する | 明確、慎重 | 破壊的操作、確認画面 |

### 統一ルール

- **一画面内で形式を統一する**。「保存」と「キャンセルする」が並ぶのは不統一
- プライマリボタンとセカンダリボタンで形式を変えない
- 破壊的操作は文脈付きにして明確にするのは例外として許容

### 操作種別ごとの推奨

| 操作 | 体言止め | 辞書形 | 推奨 |
|---|---|---|---|
| 主要操作 | 送信、保存 | 送信する、保存する | どちらでもよい |
| 破壊的操作 | 削除 | 削除する / このプロジェクトを削除する | 文脈付きが安全 |
| キャンセル | キャンセル | キャンセルする | 体言止めが多い |
| 戻り | 戻る | — | 辞書形が自然 |
| 次へ | 次へ | 次へ進む | 「次へ」が定番 |

### やってはいけないこと

- ボタンに敬語を入れない（「ご送信ください」はボタンラベルとして不自然）
- 英語の直訳をしない（「Submit」→「サブミット」ではなく「送信する」）
- 長すぎるラベル（3〜8文字が理想。10文字超えは要検討）

## 3. エラーメッセージ

### 構造: 原因 + 対処法

エラーメッセージは「何が起きたか」と「どうすればいいか」の2部構成にする。

```
❌ 「エラーが発生しました」
   → 原因も対処法もない

⚠️ 「メールアドレスが不正です」
   → 原因はあるが対処法がない。「不正」は攻撃的

✅ 「メールアドレスの形式を確認してください（例: name@example.com）」
   → 原因（形式が違う）+ 対処法（例を示す）
```

### 温度感の3段階

| 温度 | 例 | 使いどころ |
|---|---|---|
| 事務的 | メールアドレスは必須項目です | インライン バリデーション |
| 中立 | メールアドレスを入力してください | フォームの標準エラー |
| 寄り添い | メールアドレスの入力をお忘れではありませんか？ | 重要なフォーム、コンバージョン直前 |

事務的すぎると冷たい。寄り添いすぎると冗長。プロダクトのトーンに合わせて中立を基本に調整する。

### バリデーションメッセージのパターン

| 種類 | ❌ NG | ✅ OK |
|---|---|---|
| 必須 | 必須です / 入力してください | ○○を入力してください |
| 形式 | 不正な形式です | ○○の形式で入力してください（例: ...） |
| 文字数 | 文字数が超過しています | ○○文字以内で入力してください（現在: N文字） |
| 一致 | パスワードが一致しません | パスワードが一致しません。もう一度入力してください |
| 範囲 | 範囲外です | ○○から○○の間で入力してください |

### システムエラー

```
❌ 「500 Internal Server Error」
❌ 「予期しないエラーが発生しました」

✅ 「保存できませんでした。しばらく待ってからもう一度お試しください」
✅ 「接続に問題があります。ネットワーク設定を確認してください」
```

ユーザーにとって意味のある情報を伝える。HTTPステータスコードや技術的な詳細は開発者コンソールへ。

## 4. 確認ダイアログ

### 構造: タイトル + 本文 + ボタン2つ

```
┌─────────────────────────────────────┐
│ タイトル: 何をしますか？               │
│                                     │
│ 本文: 影響の説明 + 取り消し可否        │
│                                     │
│        [キャンセル]  [実行する]        │
└─────────────────────────────────────┘
```

### 破壊的操作のパターン

```
✅ よい例:
タイトル: このプロジェクトを削除しますか？
本文: 削除すると、すべてのタスクとファイルが完全に消去されます。この操作は取り消せません。
ボタン: [キャンセル] [削除する]
```

```
❌ 悪い例:
タイトル: 確認
本文: 削除しますか？
ボタン: [はい] [いいえ]
```

### ルール

- タイトルに**具体的な対象**を含める（「確認」「注意」だけは不可）
- 本文に**影響の範囲**と**取り消し可否**を明記する
- ボタンは「はい/いいえ」ではなく**動作を示す動詞**にする
- 破壊的操作のボタンは警告色（赤系）にし、デフォルトフォーカスはキャンセル側

## 5. カタカナ語の判定基準

### 原則: 和語で通じるなら和語

カタカナ語の多用は可読性を下げる。特に一文にカタカナ語が3語以上連続すると読みにくい。

```
❌ 「プロジェクトのスケジュールをリスケジュールしてステークホルダーにリマインドする」
✅ 「プロジェクトの予定を変更して関係者に連絡する」
```

### IT/ビジネス用語として定着しているもの（カタカナ可）

以下は和語に置き換えると逆に不自然になるもの:

- **UI/技術**: アプリ、ブラウザ、ログイン、ログアウト、パスワード、メール、ファイル、フォルダ、アップロード、ダウンロード、サーバー、データ、API
- **操作**: クリック、タップ、ドラッグ、スクロール、コピー、ペースト
- **ビジネス**: プロジェクト、チーム、メンバー、ダッシュボード、レポート、フィードバック
- **状態**: オンライン、オフライン、エラー、ステータス

### 和語に置き換えるべきもの

| カタカナ | 和語 |
|---|---|
| リマインド | 通知、お知らせ |
| リスケジュール | 予定変更 |
| ステークホルダー | 関係者 |
| アジェンダ | 議題 |
| コンセンサス | 合意 |
| プライオリティ | 優先度 |
| エビデンス | 根拠、証拠 |
| ペンディング | 保留 |
| フィックス | 確定 |
| アサイン | 割り当て |

### UIでの判断基準

1. ユーザーがその用語を日常的に使っているか？
2. 和語に置き換えて意味が明確に伝わるか？
3. 既存UIで使われている用語との整合性は？

3つとも「はい」なら和語を使う。技術者向けツールではカタカナ語の許容度が高い。一般消費者向けでは和語を優先する。

## 6. 主語・目的語の明示

### 省略してよい場面

日本語は主語を省略しても文法的に成立する。UIでも省略が自然な場合がある。

- 「保存しました」→ ユーザーが今操作した直後。何を保存したか文脈で明らか
- 「送信されました」→ フォーム送信直後のトースト。文脈で明らか

### 明示すべき場面

文脈がない、または複数の対象が考えられる場合は明示する。

```
❌ 「削除しました」→ 何が消えたのか不明（タスク？ファイル？プロジェクト？）
✅ 「タスク "デザインレビュー" を削除しました」

❌ 「更新されました」→ 何が更新された？ 誰が更新した？
✅ 「プロフィールを更新しました」

❌ 「権限がありません」→ 誰の？ 何に対して？
✅ 「このファイルを編集する権限がありません。管理者に連絡してください」
```

### システム主語 vs ユーザー主語

- **ユーザーの操作結果**: 能動態。「保存しました」「送信しました」
- **システムの動作**: 受動態 or 「〜されました」。「メールが送信されました」「バックアップが完了しました」
- 混在に注意: 「保存しました。メールが送信されました」→ 主語が切り替わっている

## 7. 読点の打ち方

### 基本ルール

- **修飾語が長いとき**、係り先の前に読点を打つ
- **並列する要素が3つ以上**あるとき、要素の間に読点を打つ
- **接続詞の後**に読点を打つ（ただし、しかし、また）

### UIコピーでの注意

UIコピーは短いので読点が不要な場合も多い。無理に打つと却って読みにくくなる。

```
⚠️ 「この、操作は、取り消せません」→ 読点が多すぎる
✅ 「この操作は取り消せません」

⚠️ 「名前とメールアドレスと電話番号を入力してください」→ 読点なし3つ並列
✅ 「名前、メールアドレス、電話番号を入力してください」
```

### 有名な曖昧文

「ここではきものを脱いでください」
- 「ここで、はきものを脱いでください」（履き物）
- 「ここでは、きものを脱いでください」（着物）

読点の位置で意味が変わる。UIでは絶対に曖昧さを残さない。

## 8. 数値・単位の表記

### 基本ルール

- 数字は**半角**を使う（全角数字はUIでは避ける）
- 数字と単位の間に**半角スペースを入れない**（日本語の慣習）: `3日`、`5GB`、`¥1,980`
- 4桁以上の数値には**カンマ区切り**: `1,000`、`10,000`

### フォーマット一覧

| 種類 | ✅ 推奨 | ❌ 避ける |
|---|---|---|
| 日付 | 2024年3月15日 / 2024/03/15 | 2024-3-15、三月十五日 |
| 時刻 | 14:30 / 午後2時30分 | PM 2:30 |
| 金額 | ¥1,980 / 1,980円 | ￥1980、1980 円 |
| パーセント | 87% | 87 %、８７％ |
| 容量 | 5GB / 500MB | 5 GB |
| 件数 | 3件 / 10個 | 3 件 |

### 一貫性

一つのプロダクト内でフォーマットを統一する。「¥1,980」と「1980円」が混在すると表記ゆれ。

## 9. 文体の基本ルール

UIコピーの品質を左右する、日本語の書き方の基本。

### 冗長表現を避ける

UIは簡潔さが命。回りくどい表現は排除する。

| ❌ 冗長 | ✅ 簡潔 |
|---|---|
| ○○することができます | ○○できます |
| 削除を行なう / 削除を実行する | 削除する |
| 変更を加える | 変更する |
| 確認を行ってください | 確認してください |

「〜することができる」「〜を行なう」は代表的な冗長パターン。動詞一語で済むなら動詞一語にする。

### 格助詞を省略しない

「が」「の」「を」「に」「へ」「と」「から」「で」などの格助詞を省くと、語句の関係が曖昧になる。

```
❌ 「項目名変更します」
✅ 「項目名を変更します」

❌ 「操作完了したら〜」
✅ 「操作が完了したら〜」
```

UIコピーは短いから助詞を省きたくなるが、省くと意味が不明瞭になる。ボタンラベル（体言止め）は例外。

### 漢字とひらがなのバランス

漢字が多すぎると堅く読みにくい。ひらがなが多すぎると意味の区切りが分かりにくい。

**ひらがなにしたほうが読みやすい語**

| 漢字 | ひらがな |
|---|---|
| 下さい | ください |
| 致します | いたします |
| 頂く | いただく |
| 出来る | できる |
| 予め | あらかじめ |
| 但し | ただし |
| 及び | および |
| 又は | または |
| 殆ど | ほとんど |
| 既に | すでに |

補助動詞（「〜してください」「〜していただく」「〜してみる」）はひらがなで書く。

### 常用漢字の範囲で書く

常用漢字表にない読み方（表外読み）は避ける。

| ❌ 表外読み | ✅ 常用漢字 |
|---|---|
| 想い | 思い |
| 旧い | 古い |
| 繋がる | つながる |
| 拘る | こだわる |

### 用語の一貫性

同じものを指す言葉を統一する。同じ画面で「設定」と「セッティング」、「削除」と「消去」が混在すると、ユーザーは別の操作だと誤解する。

- 用語集（UI用語辞書）をプロダクト内で定義する
- 一度決めた用語は全画面で統一する
- ヘルプ等で理解を促す意図的な言い換えは例外

### 一般的な言葉を使う

社内用語・業界専門用語・独自の造語は避け、辞書に載っている一般的な言葉を使う。ユーザーがその言葉を知らない可能性があれば、より平易な言葉に置き換える。

## 10. 空状態・ゼロデータのコピー

空状態はユーザーが最初に目にする画面であることが多い。冷たい空白ではなく、次のアクションへ導く。

### パターン

```
❌ 「データがありません」
❌ 「No results found」（日本語UIに英語）

✅ 構造: 状態の説明 + 次のアクション

「まだプロジェクトがありません」
「新しいプロジェクトを作成して、チームでの共同作業を始めましょう」
[プロジェクトを作成する]
```

### 場面別

| 場面 | コピー例 |
|---|---|
| 初回利用 | まだ○○がありません。○○を作成して始めましょう |
| 検索結果なし | 「○○」に一致する結果が見つかりませんでした。別のキーワードで検索してください |
| フィルター結果なし | 条件に一致する○○がありません。フィルターを変更してください |
| 権限不足 | この○○を表示する権限がありません。管理者に連絡してください |
| エラー後 | ○○を読み込めませんでした。[もう一度試す] |

### ルール

- **状態だけでなくアクションを示す**（「ありません」で終わらせない）
- **ユーザーのせいにしない**（「あなたはまだ何も〜」は避ける）
- **CTAボタンを置く**（テキストだけでなく、次の操作を物理的に提供する）

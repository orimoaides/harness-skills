# clarify — 日本語UIコピー評価モード

`/zen` のUIコピーモード。日本語Webサイト・アプリのUIコピーを評価し、具体的な改善を行う。

対象は「UIに現れる短いテキスト」 — ボタン、エラーメッセージ、フォームラベル、通知、ツールチップ、空状態、確認ダイアログ。ブログ記事やマーケティングコピーは対象外。

## ワークフロー

### Step 1: UIコピーの収集

対象が指定されている場合はそのファイルを読む。指定がなければ以下を探す:

1. ページコンポーネント（`page.tsx`, `index.tsx` 等）
2. UIコンポーネント（`components/ui/`, `components/` 配下）
3. エラーハンドリング（`error.tsx`, `not-found.tsx`, try-catch 内のメッセージ）
4. フォームコンポーネント（バリデーションメッセージを含む）
5. 定数・i18n ファイル（`messages.ts`, `constants.ts`, `locales/ja.json` 等）

収集する対象:
- ボタンラベル
- フォームラベル・プレースホルダー
- エラー・バリデーションメッセージ
- 確認ダイアログ（タイトル + 本文 + ボタン）
- トースト・通知メッセージ
- 空状態メッセージ（empty state）
- ツールチップ・ヘルプテキスト
- ナビゲーションラベル

### Step 2: 評価

[日本語UXコピー リファレンス](japanese-ux-copy.md) に基づいて評価する。

以下の各項目を ✅ / ⚠️ / ❌ で判定:

**敬語レベルの一貫性**
- 画面・セクション単位でトーンが揃っているか
- です/ます調と体言止めが混在していないか
- エラーメッセージだけ急に丁寧 or 冷たくなっていないか

**ボタンラベル**
- 動詞の活用形式が統一されているか（体言止め / 辞書形 / ます形）
- 主要アクションの意図が明確か
- 破壊的操作が慎重さを示しているか（文脈付きラベル or 確認ダイアログ）
- 長すぎないか（3〜8文字が理想）

**エラーメッセージ**
- 「原因 + 対処法」の構造になっているか
- ユーザーを責めていないか（「不正」「無効」は避ける）
- 温度感がプロダクトのトーンに合っているか
- システムエラーが技術用語でなくユーザー向けの言葉になっているか

**カタカナ語の密度**
- 和語で通じる語がカタカナに置き換えられていないか
- 一文にカタカナ語が3語以上連続していないか
- IT用語として定着したカタカナ（ログイン、ファイル等）との区別ができているか

**主語・目的語の明示**
- 「削除しました」のように「何を」が不明な文がないか
- システム主語とユーザー主語が混在していないか
- 文脈がない場所で主語を省略していないか

**読点の位置**
- 修飾語の係り先が明確か
- 並列要素に適切に読点が打たれているか
- 読点の過剰 or 不足がないか

**数値・単位の表記**
- 半角数字が使われているか
- カンマ区切り、日付・時刻・金額のフォーマットが統一されているか
- 単位との間のスペースルールが一貫しているか

**文体の基本ルール**
- 冗長表現がないか（「することができる」→「できる」）
- 格助詞が適切に使われているか（「項目名変更します」→「項目名を変更します」）
- 漢字とひらがなのバランスが適切か（補助動詞はひらがな: 「下さい」→「ください」）
- 用語が一貫しているか（同じ意味に同じ言葉を使っているか）

**空状態・ゼロデータ**
- 「データがありません」で終わっていないか
- 次のアクションへの導線があるか
- ユーザーのせいにしていないか

### Step 3: 報告

スコア算出: 評価カテゴリは **計 9 項目**（敬語レベル / ボタンラベル / エラーメッセージ / カタカナ語の密度 / 主語・目的語の明示 / 読点の位置 / 数値・単位の表記 / 文体の基本ルール / 空状態・ゼロデータ）。各カテゴリを ✅=1 / ⚠️=0.5 / ❌=0 で合算（小数点第一位まで）。該当要素が画面に存在しないカテゴリ（例: ボタン不在ページの「ボタンラベル」、エラー表示不在ページの「エラーメッセージ」）は分母から除外し、表示分母を調整（例: `X / 8`）。デフォルト表記は `X / 9`。

評価結果を以下の形式で報告する:

```
## 日本語UXコピー評価

### スコア: X / 9

✅ 敬語レベル: です/ます調で統一されている
⚠️ エラーメッセージ: 「○○は必須です」— 対処法が示されていない
❌ ボタンラベル: 「送信」「キャンセルする」「戻る」— 活用形が混在
...

### 改善提案（優先度順）

1. **[高]** ボタンラベルの統一
   - before: 「送信」「キャンセルする」「戻る」
   - after:  「送信する」「キャンセルする」「戻る」

2. **[中]** エラーメッセージに対処法を追加
   - before: 「メールアドレスは必須です」
   - after:  「メールアドレスを入力してください」

3. **[低]** ...
```

### Step 4: 修正

ユーザーが修正を求めた場合、具体的なテキスト変更を提案・実行する。

変更時の原則:
- プロダクトの既存トーン（フォーマル/カジュアル）を尊重する
- 一画面内の全コピーを見て一貫性を確保する
- 変更前後でどうユーザー体験が変わるかを説明する
- i18n ファイルがある場合はそこを修正する
- ハードコードされた文字列は定数化を提案する

## やらないこと

- 長文コンテンツの推敲（ブログ記事、利用規約等）
- マーケティングコピー（キャッチコピー、LPの訴求文）
- タイポグラフィの評価（typeset モードの領域）
- 配色・レイアウトの評価
- 多言語対応のアーキテクチャ設計

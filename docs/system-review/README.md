# system-review — システム自体を監査する評価者パネル

バナーを評価するのではなく、**この評価システム自体（スキル7本＋思想＋運用）の漏れ・穴を審査する**4人の評価者。それぞれ別の利害を代表するので、同じシステムを見ても違う欠陥が見つかる。

| 評価者 | 代表する利害 | 見つける欠陥タイプ |
|---|---|---|
| [operator.md](operator.md) 作業者 | 日々回す人の手数と迷い | 摩擦・手戻り・「手でやった方が早い」ポイント |
| [admin.md](admin.md) システム管理者 | 複数人運用・保守・配布 | 運用事故・データ腐敗・属人化 |
| [senior-designer.md](senior-designer.md) シニアデザイナー | 評価の質と現場の判断の一致 | 評価と実務の乖離・デザイン文化への副作用 |
| [ai-expert.md](ai-expert.md) AI専門家 | LLM/エージェント工学の精度と安全 | 起動混線・バイアス・コスト・injection |

## 共通の出力フォーマット（全員これで返す — 統合・重複排除のため）

```json
{ "findings": [ {
  "id": "OP-1",
  "reviewer": "operator|admin|senior|ai",
  "target": "対象ファイル or 工程名",
  "issue": "欠陥の1文",
  "severity": "high|mid|low",
  "evidence": "根拠（該当箇所の引用 or 具体シナリオ）",
  "proposal": "直し方の案（1行）"
} ] }
```

## 共通ルール
- 審査対象はリポジトリ全体（skills/・docs/・README）。**褒めない** — 見つけた欠陥だけ返す。
- 憶測の欠陥は禁止。evidenceに実在の記述か具体的な失敗シナリオを必ず書く。
- 各自、自分の観点リスト外でも気づいた欠陥は「対象外だが」と断って報告してよい。
- 実行はサブエージェント推奨（各評価者に自分のカード＋リポジトリを渡し、互いの結果は渡さない）。討論のさせ方（独立のみ／独立→相互批判→裁定／段階式）は人間が決める。

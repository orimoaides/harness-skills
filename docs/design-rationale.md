# design-rationale — 設計判断の根拠（長文版）

SKILL.md本文はセッション中コンテキストに常駐するため、本文には行動を守る1行理由だけを残し、長い設計論はここに退避する。

## なぜ class A / B / C の三分類か
分類の軸は**重要度ではなく測定可能性**。重要度は配点で表す。この2軸を混ぜると「大事だけど測れない観点」（例: カーニング）の扱いを誤る — 苦手だから配点を下げると「文字組が甘くても高得点」のシステムになり、人間FBと永遠にズレ続ける。

## なぜ機械照合をスクリプトにするか
LLMが画像を目視してtrue/falseを言うのは機械照合ではない。寸法・比率・容量・形式はコードで測れる（check_image.py）。測れるものを測らないと「照合できるものだけがブロック対象」という原則の根拠が崩れる。クラシルのlint検証（一発動作率20%→100%）が先行実証。

## なぜ10段階で「帯が先」か
1点単位の一致は人間同士でも無理。帯（1–3致命/4–7要改善/8–10合格）を先に決めれば、較正は帯一致で収束可能になり、帯内の1点刻みは改善の勾配情報として残せる。

## なぜ影スコア（class Cにも判定させる）か
判定を出させないと較正データが永遠に貯まらず、苦手観点が苦手なまま固定される。一方で集計に入れると、精度が育つ前の不安定な判定が点数全体の信用を壊す。「出すが数えない→帯一致が安定したら数える（B昇格）」が両立解。

## なぜ独立評価はサブエージェント必須か
1つのコンテキストで5役を順に演じると、後の役に前の役の判定が見える＝アンカリングされ、5役が実質1役になる。独立性は指示ではなく実行構造（並列サブエージェント）でしか担保できない。

## なぜ点数はAIに発明させないか
点数の価値は人間FBとの差分を測れることにある。AIが感覚で合計を丸めた瞬間、その点は較正に使えなくなる。判定（帯・スコア）はAI、集計は算術、配点の決定は人間。

## なぜ理由型で書くか
命令型（MUST/NEVER）は制約衝突時に一方を黙って破棄し捏造率51%、理由型は0%という実証（classmethod, 2026）。理由があるとAIは目的の別解を自力で構築できる。公式ドキュメントの「本文は簡潔に」との折衷として、本文には判定を歪めうる箇所の1行理由のみ残す。

## なぜ承認ゲートを人間に固定するか
AIが自分の採点基準を自分で書き換えられると、較正が「人間に合わせる」ではなく「自分の過去の判定に合わせる」方向に閉じる。ルールの増えすぎ（ノイズ制約化）も人間にしか止められない。

# ops-loop — 評価ループ運用の掟

出典: Codez「Loop engineering: 14-step roadmap」(2026-06、Anthropic技術文書＋Addy Osmani論考ベース) から、この評価ハーネスに適用できる原則を移植したもの。

## 1. 北極星の指標: 「採用された指摘率」
ループの成否はトークン消費でも指摘の数でもなく **cost per accepted change** で測る。うちの翻訳:
> **人間が実際に採用した指摘（fix_orderのうち直された項目）の割合。50%を下回るなら、評価ループは人間の時間を節約せず消費している。**
較正記録に `adopted_fixes`（提示した改善指示のうち採用された数/総数）を記録し、帯一致率と並ぶ第2のKPIとする。

## 2. 状態ファイル — 「エージェントは忘れる。ファイルは忘れない」
運用リポジトリ直下に `STATE.md` を置き、毎回の評価・較正後に更新する:
```markdown
# 評価ループ STATE
## 最終実行
YYYY-MM-DD 案件X FB-n を評価（craft nn・分業モード）
## 進行中
- 案件X: 人間FB待ち（較正未実施）
## エスカレーション（人間の判断待ち）
- ...
## 教訓（チャットに書かずここに書く）
- YYYY-MM-DD: チャット貼り付け画像はファイル実体が無く実測不能。案件はファイルパスで受ける
## 発動した較正・昇格降格
- ...
```
理由: 教訓を会話に置くと次のセッションで消える。STATE.mdがあればループは「再開」でき、無ければ毎回「最初から」になる。

## 3. 客観ゲート優先 — 「楽観主義者が2人に増えただけ」を避ける
> No real verifier. Just a second agent asked to "review" — two optimists agreeing.（Ralph Wiggumループ）
役を増やすより客観ゲート（スクリプト実測・スキーマ検証）が先。同一モデルの専門家役は「分割と遮断」には効くが「検証」にはならない。検証はテスト・実測・人間だけ。

## 4. ゲートは腐る — 月1の「ゲート点検」
> Gates rot. 通したPRを承認したテストが、本当にその失敗を捕まえるか時々確かめろ。
うちの翻訳: **月1回、わざと壊れた入力（サイズ違い・低コントラスト画像など既知NG）を check_image.py 等に食わせ、ちゃんと false が出ることを確認する。** 実測スクリプトが静かに壊れていたら、A照合の信頼が幻になる。

## 5. 自動化の順序 — 飛ばした段の分だけ後で払う
> 手動実行を安定させる → スキル化する → ループに包む → スケジュールする。順序を飛ばすな。
現在地: 手動実行（ドライラン）を安定させる段。較正が数件回って評価が信頼できるまで、定期実行・自動トリアージには進まない。

## 6. 停止条件と予算
- 討論（panel）は1案件あたり上限回数を決めて起動する（無限に深掘りしない）
- 「done」をAIの自己申告にしない — 較正の突き合わせと人間の合否だけが完了条件

## 7. セキュリティ税
- **スキルは導入前にソースを読む**（監査された17,022スキル中520が資格情報を漏らすという計測がある。うちはmelta-ui/zen導入時に全ファイル確認済み — この運用を標準にする）
- 権限（allowed-tools）は30日ごとに再監査
- ループのログに秘匿情報を流さない（評価JSONに案件の機密を書きすぎない）

## 8. 手戻りには2種類ある — ゼロを目指さない（LayerX輸入）
- **防げる手戻り**: ブリーフの穴・矛盾・規定不備など前工程の欠陥由来。brief-dig と A照合の管轄 — ここはゼロを目指す
- **健全な手戻り**: 見て初めてわかる判断・探索のやり直し・**正本（brand-profile・基準文）の拡張につながるFAIL** — これは学習であって失敗ではない。較正の「原因束ね」で手戻りを分類するとき、この2種を区別して記録する（防げる系→前工程の修理提案、健全系→正本拡張の種）

## 9. 理解の負債 — ループが良くなるほど効く毒
- **人間はレポートを読むだけでなく、稿そのものを自分の目で見続ける**（点数を追いかける癖はループの副作用。影スコアのブラインド運用はこのための装置）
- ループには判断仕事を渡さない: 探索（何を作るか）・ブランドの方向性はループ外。ループは機械照合と定型検証の領分

# 会社版 banner-harness の思想整理（2026-07-21 スクショ取り込み）

> 折茂が会社側（別環境）で制作中の「バナー評価ハーネス」のスクショ6枚から思想を復元・整理したもの。
> **これは忘れてはいけない中核資産。** 個人版（おりもラボ）・今後のスキル化はこの思想と整合させる。
> 原本: 会社側 `banner-harness/harness-map.md`（2026-07-15 すり合わせMTGのたたき台）

## 0. 中心思想（折茂の言葉・2026-07-21）

> **AIが作ったものを人だけで見るのではなく、「何を良しとするか」「どう検証するか」をSkillsに残す。作るAIだけでなく、評価するAIまで含めて設計する。**

- 人間のレビューは消えない。ただし人間の目にしか無かった「良しの基準」と「検証の手順」を資産（Skills）に変換し続ける
- システムは三層: **作るAI**（生成）／**評価するAI**（スキル棚・討論モデル）／**評価するAIを較正する人間**（突き合わせ・昇格降格・配点の決定）
- 判断の言語化そのものが納品物。デザイナーの中心スキルは「正確な動作指定」から「良い状態の定義」へ（NN/G・kgsiと同方向）

## 1. 全体骨格 — 4要素のループ

デザインハーネスの定義（①制約する ②コンテキストを渡す ③検証する ④フィードバックする）にバナー評価を当てはめる。参照フレームは「リングモデル」（Tokens/Components/Patterns → Principles/Users/Existing screens → A11y/Overflow/Flow/Cognitive load → Future図）。

```
①制約する(true/false) → ②コンテキスト(判断材料) → ③検証する(スキル棚) → ④評価FB(較正)
        ↑←←←←←←←←←← 較正して次のループへ反映 ←←←←←←←←←←↓
```

## 2. スキルのクラス分け（★この設計が発明の核）

**スキル＝「①の制約」または「②の原則」を1観点ずつ判定器にしたもの**（現在21スキル）。

| class | 数 | 判定 | 効力 |
|---|---|---|---|
| **A** | 7 | 機械照合 true/false | **1つでもfalseならブロック。意図宣言でも覆らない**（変更は案件ファイルへの記録で行う） |
| **B** | 14(暫定) | 基準＋コンテキストで定性判定（3段階） | Fail=減点。意図宣言で不問にできる |
| **C** | — | 人間領域。AIは観察事実の列挙まで | **較正の結果で降格して作る**（最初からは作らない） |

- **class A 7スキル（2026-07-15 MTG確定、frontmatter `class:` 反映済み）**: `req-mandatory` / `req-text-accuracy` / `req-ng`（位置・存在系）/ `typo-line-break`（禁則系）/ `output-size-spec` / `output-multi-size` / `output-logo`
- 判定の性質: **機械照合できるもの（true/false）だけをAに置く**。「目立ちすぎ」のような程度問題は制約ではなくBの原則へ
- 混在スキル（req-ng, typo-line-break, output-multi-size）は A項目=判定基準、B項目=「参考観点」として本文内で分離
- 案件ごとに評価セットを選択（全部盛りにしない）、フェーズ（FB-2〜6）でさらに絞る

## 3. ①制約する — 破ったらブロック。交渉の余地なし
| 制約 | 出どころ | 状態 |
|---|---|---|
| 必須要素（日時・氏名・肩書き・ロゴ等の存在と正確性） | 依頼書（案件ごと） | ✅ extract で自動抽出 |
| 文字の正確性（誤字・副題の無断変更） | 依頼書 | ✅ req-text-accuracy 起票済み |
| NG事項・序列制約（「登壇者より上NG」等） | 依頼書 | ✅ 同上 |
| 改行・禁則（行頭禁則・日時の行またぎ） | 組版ルール | ✅ typo-line-break をA化 |
| 自社ロゴレギュレーション | ブランド規定 | □ 箱作成済み(output-logo)。規定は要提供 |
| サイズ・容量・入稿形式 | 入稿規定 | □ 箱あり(output-size-spec)。規定は要提供 |

## 4. ②コンテキストを渡す — 判断の材料。多いほど良いわけではない
- Principles: デザイン原則（各スキルの判定基準文）✅スキル棚に内蔵
- Users: ターゲット・訴求優先順位・トーン判断軸 ✅案件ファイルに抽出
- Existing screens: 過去案件の最終FIX（良例）・NG例 ❌**空。最重要の未収集資産**
- レビュー文化: シニアの指摘傾向（FBログ）✅箱はある。中身はこれから
- **設計方針: 評価のたびに全部渡さない。フェーズ該当スキル＋案件ファイルだけ読む（ノイズはコンテキストの敵）**

## 5. ④フィードバックする — 失敗のたびルールと検証を1つ追加
- シニアFB×AIレポートの突き合わせ → 一致/見落とし/誤指摘 → スキルの精度カウンタ＋FBログ更新
- **見落とし2回 → 基準追加 or 新スキル**（=検証が1つ増える）
- **誤指摘2回 → 基準を絞る**（=ルールが正確になる）
- どのスキルにも該当しない指摘 → 新スキル起票
- 突き合わせ記録は `calibration/`（案件個別）

## 6. 2層構造 — 共通基盤とドメイン層
- **共通基盤: `marketing/harness-skill-builder/` プラグイン（2026-07-15新設）**
  - `harness-skill-builder`: スキル/プラグイン開発の下地6ステップ（要件深堀り→機械的制約の設定→分割→制作→自己検証→FBフォーム案内）。**今後スキルを作る・直すときは必ずこれを通す**
  - `skill-accessibility-eval`: **スキルを評価するスキル**。8カテゴリ（トリガー精度/出力一貫性/トークン効率/構造健全性/安全性/評価可能性/堅牢性/記述スタイル）。乖離データが貯まったら banner-harness のスキル自体を採点・改修する自己採点ループにつなぐ
- **ドメイン層: `banner-harness`（21スキル）** — 上記6カテゴリ

## 9. 取り込み: PKSHA記事（zenn/pksha 清水はるか「ヒューリスティック評価のスキル化」）

banner-harness に直接効く知見:
- **前提の置き方が同じ**: AI評価精度は30〜40% → 「最低限のガードレール」と位置づけ、人間との役割分担を明示（=折茂の較正思想・class C「人間領域」と一致）
- **universal / contextual の項目分離**（全画面必須 vs 条件依存）→ banner-harness の「評価セットを案件ごとに選択・フェーズで絞る」と同型。分離の粒度の参考になる
- **未確認項目の可視化**: 「確認したインタラクション状態」テーブルで*何を検証していないか＋理由*を出力させる → banner-harness の測定手段明記（実測/目視/読解）に「未確認テーブル」を足す価値あり
- **未確認理由のループバック**: 一度AIに失敗をさせてから最後に「操作未実施」だけ再実施 → **AIのサボり対策**。banner-harnessに未実装の観点
- **評価項目と結果を同じ行に並列表示** → AIの解釈ズレを可視化（較正の突き合わせが楽になる）
- インプット4形式（コード/スクショ/ライブ/混在）ごとのワークフロー事前定義
- 構成: SKILL.md＋references/ 分割、7ステップ化。成果は評価時間約1/10
- 記事: https://zenn.dev/pksha/articles/8f4d45b913ed50

## 11. 追加スクショ取り込み（2026-07-21 第2弾）

### 正準の4層定義（折茂本人の言葉。これを正とする）
1. **① 文脈**: AIに「何を使うべきか」を渡す
2. **② 制約**: AIに「何をやってはいけないか」を渡す
3. **③ 検証**: ルールと実物のズレを自動で見つける
4. **④ フィードバック**: 人間の修正を、次のルールに戻す

### メタスキルのループ（ホワイトボード構想）
スキル自体を運用する4つのメタ部品が循環する:
```
skillsを選択するskills ⇄ 検証・品質チェックするskills ⇄ そのskillsを評価データから更新するskills ⇄ 評価データ
```
- **選択**: 案件・フェーズに合わせて評価セットを選ぶ（全部盛りにしない）
- **検証・品質チェック**: スキル棚で実際に判定する実行体
- **評価データ**: 実走結果・シニアFB突き合わせの蓄積（calibration/）
- **更新**: 評価データが入るたびにスキル棚自体を改修する
→ **このシステムは「評価をしまくって、データが入るたびに改善される」自己改善システムになる**（折茂の設計意図）

### A制約ボード（具体票）
「trueとfalseレベルの物」＝破ったらブロック。機械照合できるものだけ。意図宣言でも覆らない:
- 納品形式 / 画像サイズの確認（比率）→ output-size-spec
- 文字の正確性 → 文字の間違い（アクセシビリティのガイドで明文化されていたら制約）・副題の変更など → req-text-accuracy
- 改行・禁則 → typo-line-break
- 要件チェック / 必須要素の照合 → req-mandatory
- 自社ロゴの規定 → output-logo
- 画像/素材品質 →（照合可能な部分のみA、程度問題はphoto系のBへ）
